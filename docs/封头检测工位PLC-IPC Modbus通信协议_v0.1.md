# 封头检测工位 PLC-IPC 通讯协议

文档版本：v0.1
编制日期：2026-04-01  
协议定位：PLC 、IPC 专用通讯协议

## 1. 编制依据

本通讯协议的编制立足于实际工程部署要求，深度参考了设备的物理拓扑、电气对接能力以及算法流程约束。具体的指导和参考依据包括但不限于：

1. **项目需求及前期草案**：`plc_modbus.docx`与早期地址定义 `plc_modbusl.xlsx`，构成了初始的寄存器空间规划框架。
2. **工艺与设备动作流则**：`1 设备工作说明.docx` 以及 `工作流程.vsdx` 第 1 页“封头检测工位”的详细说明。
3. **系统联动逻辑推演**：提取了工位所需的联锁避碰、上下料引导、多相机视觉检测协同等时序需求。

*注：本通讯协议主要聚焦于 PLC 与工控机（IPC）交互接口部分的标准化。如在后续现场调试中设备执行节拍、机械臂运动逻辑及 AI 算法判定策略发生调整，本协议（含相关握手时序和地址表）需同步进行状态重演和升版修订。*

## 2. 适用范围

本协议用于明确并约束 **封头检测工位** 核心控制设备（即 PLC 与边缘计算工控机 IPC）之间的通讯数据交换边界及交互准则。

### 2.1 适用对象边界
- **本协议覆盖范围**：
  - PLC：负责向 IPC 发送模式、阶段控制指令、物理状态触发以及动作需求（如：上料解算、分段扫描协同等）。
  - IPC (视觉处理与总控)：负责解析指令，协调各类 2D/3D 相机采集与轨迹计算，回送空间坐标、检测定性结果（良/疵品）及软件层环境安保状态。
- **本协议不覆盖范围**：
  - PLC 直接操作底层驱动硬件（如气缸、电机、外部变频器）的协议指令。
  - IPC 内部各个子进程或子传感器模块（如防碰撞雷达硬件、MES 后端网关）的自身链路协议。

### 2.2 协议功能涵盖清单
系统内依托于本 Modbus 映射区域覆盖的核心功能包括：
1. **全局安全监控**（雷达防撞预警、AI 监控联动及人员侵入禁停）。
2. **设备维护测试**（设备软硬件状态巡检及在线心跳）。
3. **视觉检测全业务流**（上下料判断、识别解算抓取点、通孔与位姿核验、连续网格扫描配合、OCR 识别执行及下料判断等）。
4. **异常流转处理**（异常报警接驳释放、复位及残存记录冲洗清空）。

## 3. 通讯基础约定

本章定义了链路配置规范及数据收发的基础同步法则，旨在为后续特定寄存器的读写与动作时序（详见第 5、6 章）提供底层的架构标准。

### 3.1 通讯底座与软硬件角色
- **协议类型**：Modbus TCP（以太网接口）。
- **服务端 (Server)**：全由 **PLC** 承担。所有的保持寄存器（Hold Registers, `40001` - `4xxxx`）均由 PLC 一次性开辟并暴露。
- **客户端 (Client)**：由 **IPC (工控机)** 充当。主动建立连接并以轮询或突发请求执行写入、读取工作。
- **通讯端口/ID**：端口通常固定为 `502`；从站 ID 默认 `1`（按实际网络规划修正）。

### 3.2 数据格式与存放规律
为了避免跨平台内存对齐问题和乱码现象，本系统限定使用如下基础类型：
- `INT16` / `UINT16`：占据单寄存器（1个 Word=2字节），分别做有符号或无符号映射，状态码多数使用无符号 `UINT16`。
- `DINT32` / `FLOAT32`：跨 2 个连续的寄存器。特别注意 `FLOAT32` 在本系统通常建议采用 **`CDAB`** 字序（大端序结合内部 16-bit 小端翻转表示），以应对常规主流自动化设备的浮点对齐问题。
- `ASCII[n]`：按每寄存器 2 字符存放，以高字节先存的方式表达字符串（主要用于扫码编号输出）。

### 3.3 交互频率及实时性（轮询与超时设限）
为兼顾业务响应性与网卡负荷，制定如下交互准则：
- **基带扫描与心跳**：主干状态（PLC 系统模式 `PLC_SystemState`，各安防位 `Safety_Status_Word` 等）IPC 必须在 `50~100ms` 固定周期内读取及同步自身对应的 IPC 反馈端状态。
- **在线监测判定**：心跳寄存器（如 IPC_Heartbeat）每 1000ms 更新。收发方如连续 `3秒` 观测到对端数据不刷新，判为断联。
- **指令过程超时警戒线（建议值）**：
  - 上料抓取点解算及下料寻址：10 秒
  - 定点通孔和位姿快速复核：5 秒
  - 单网格点云拼接组装或计算汇总：60 秒
  *(超时阈值为通信状态兜底，达到阈值将抛出 `Res=6` 超时处理码终止锁死)*

## 4. 设计原则

本协议架构不限于单点收发，更强调工艺控制维度的可靠流转。各项寄存器的排布、状态位的制定及编码规则秉持以下基本原则：

### 4.1 状态解耦原则
系统严格限定数据产生端的写权限，避免出现数据冲刷：
- **命令与控制 (40001~40100)**：专门预留作为 PLC 发下指令（`Cmd/Trig_xxx`）及输出机械反馈结果，该区域对 IPC 为**只读**。
- **应答与汇报 (40101~40200)**：专门留给 IPC 用以写入自查状态、回写判定结果和坐标反馈（`Ack/Res_xxx`），该区域对 PLC 只做接收读取用。（见地址表分类）。
从而实现通信全双工互不干涉。

### 4.2 四步握手执行原则
面对现场极其复杂的多相机与扫描联动时序，针对任何过程级的启动事件，都必须通过“触发请求”→“收到执效”→“结果回写”→“完成脱钩”的闭环状态机来完成（结合第 6 章应用与第 5.3 节定义）：
1. **[触发]** PLC 发送触发号 `Trig_xxx = 1`。
2. **[锁定]** IPC 探测跳变后迅速写入 `Ack_xxx = 1`，PLC 借此得知指令生效，不允许重复或并发覆盖此信号。
3. **[回填]** IPC 后台业务完结时，按次序**先落库具体测量结果或计算坐标**，**随后提交业务结果码 `Res_xxx`**，**最终挂出 `Ack_xxx = 2(或3)`。**这种顺序避免 PLC 提前因探测到状态结束而去拿到了旧空包数据。
4. **[释放]** PLC 取走数据后清除 `Trig_xxx = 0`，等待 IPC 回复 `Ack_xxx = 0`，完成一轮业务清理。

### 4.3 保护与中断恢复原则
- **不间断安全监控**：当 IPC 检测防碰预警越限或异常落位（如报警等级达橙黄甚至深红区）时，不论此时任务环流处在何地，须第一时间置激安全寄存器位（`IPC_SafetyAction_Word` / `Device_Fault`）。底层控制响应此指令后具有“一票否决”中断运行的优先级（参考 6.1 与 6.10 节）。
- **残渣数据强洗涤**：任何因严重警报、断电引起的中断跌出，系统不得带伤复工。二次流转前强制插入 `Trig_ResultReset` 并以人工或探测器排查场区残留判定结合执行（遵守时序 6.10 约束规范）。

### 4.4 的参数命名延伸原则
- 为杜绝地址表的列名分化、中英文命名混淆，Markdown 时序文档中的所有重要状态变量名及其代码标识，已高度对齐到基础地址总表。

## 5. 状态与编码定义

本章节结合第 6 章“关键时序说明”和地址表中的规划，系统定义了设备在各模式、阶段、指令及结果反馈中的编码规范。

### 5.1 运行模式与控制状态编码

该部分用于定义设备全局运行的工作模式与系统状态。

#### 5.1.1 PLC 系统与工作模式
- **`PLC_SystemState` (PLC系统状态)**
  - `0`：待机 (Standby)
  - `1`：自动运行 (Auto)
  - `2`：暂停 (Pause)
  - `3`：报警停机 (Alarm/Stop)
  - `4`：手动 (Manual)
- **`Station_WorkMode` (工位工作模式)**
  - `0`：空闲
  - `1`：上料模式
  - `2`：扫描检测模式
  - `3`：检测演算模式
  - `4`：下料模式
- **`Flow_Enable` (流程允许运行)**
  - `0`：禁止运行（存在安全触发、人员未撤离或自检未通过等防呆情况）
  - `1`：允许运行

#### 5.1.2 IPC 系统状态 (IPC -> PLC)
- **`IPC_SystemState` (IPC 整体状态)**
  - `0`：未初始化
  - `1`：初始化中
  - `2`：就绪 (Ready)
  - `3`：忙碌 (Busy)
  - `4`：暂停
  - `5`：故障 (Fault)
- **`IPC_Ready` (IPC 是否接受新任务)**
  - `0`：否
  - `1`：是

### 5.2 流程阶段与执行状态编码

定义 IPC 当前处理业务处于的具体进度和阶段。

- **`IPC_CurrentStage` (当前处理阶段)**
  - `0`：空闲 (等待触发)
  - `1`：安全与环测监控 (常态常驻)
  - `2`：设备初始自检
  - `3`：上料工位有料判断
  - `4`：上料抓取点 3D 解算
  - `5`：扫描转台有料判断
  - `6`：通孔/起偏位姿校验
  - `7`：执行多阶段跟随扫描 (2D/3D融合)
  - `8`：AI 缺陷与 OCR 检测汇总计算
  - `9`：下料区满料判定及卸料点解算
  - `10`：结果归档与缓冲清理断点恢复

### 5.3 交互阶段与握手应答编码 (Ack/Trig)

所有以 `Trig` 触发的指令交互（含上料、检测、设备清理等）遵循以下通用指令和应答规范：

- **系统发令/触发控制 (`Trig_xxx` / `Cmd_xxx`)**
  - `0`：复位 / 无效指令
  - `1`：触发 / 使能启动

- **系统应答握手字 (`Ack_xxx` 通用)**
  - `0`：空闲 (未触发或已复位回收)
  - `1`：已接收指令，执行中 (响应触发沿)
  - `2`：执行完成，数据已落盘，PLC可取结果
  - `3`：执行失败 (参数错等通信层拒接)

### 5.4 业务检测与执行结果编码 (Res)

各业务 `Trig_xxx` 完成后（当 `Ack=2` 时），需搭配 `Res_xxx` 告知 PLC 该项业务的终态判定结果。本系统区分基础通用码和特化业务结果码：

#### 5.4.1 通用基础结果码规范
- `0`：无效 / 未执行
- `1`：执行成功 / 判定合格
- `6`：处理超时 (底层硬件卡死或算法死循环)
- `7`：算法服务/硬件解算失败

#### 5.4.2 场景特化结果码定义
针对 6 大关键工作时序环节，`Res_xxx` 具有针对性枚举映射：
- **`Res_LoadGrasp` (上料抓取点结果)**
  - `1`：成功 (输出 X~Rz 坐标)
  - `3`：无料
  - `5`：相关设备未就绪 (例如相机脱机)
  - `9`：安全联锁触发动作中断
- **`Res_StationMaterialCheck` (扫描工位有料判断结果)**
  - `1`：有料 (应继续扫描环节)
  - `2`：无料 (要求复判或拉响报错)
- **`Res_PoseCheck` (通孔/位姿校验结果)**
  - `1`：姿态正确/可夹持
  - `2`：定位错误 (放置歪斜超限，要求人工排故)
- **`Res_ScanSegment` (单段扫描结果)**
  - `1`：本段采集完成并存组
  - `2`：条件不满足 (外部机械防碰撞干涉预警发生等)
  - `5`：跟踪或拍照光源设备未就绪
- **`Res_Inspection` (综合检测与缺陷判定)**
  - `0`：未判定
  - `1`：OK (优良品，尺寸与 AI 表面检测均合格)
  - `2`：NG (次品/废品，通过 `NG_Reason` 输显具体缺陷类型)
  - `3`：要求人工复检
- **`Res_UnloadCalc` (下料与落放点评判结果)**
  - `1`：可安全下料，已输出空间降落坐标
  - `2`：满料仓极危 (无落放位区，强制急需清理拉走)
- **`Res_SelfCheck` (多相机状态自检结果)**
  - `1`：自检通过 (校验范围合理)
  - `2`：未通过 (标定外参偏出公差警戒线，报警求维护)
- **`Res_CodeRead` (OCR 及编号扫描结果)**
  - `1`：成功 (附送读出的 ASCII 编号明文字)
  - `2`：OCR识别失败 / 模糊无解
  - `3`：需人工手动介入持枪扫码
- **`Res_ResultReset` (结果/缓存清理复位应答)**
  - `1`：成功排空当前缓存，可切入全新循环
  - `2`：空台无需处理，直接下一环

### 5.5 异常、报警与安全防呆编码

负责标识系统中枢评测的安全、软件或通信故障预警等级：

- **`IPC_AlarmLevel` (当前警告等级反馈)**
  - `0`：正常、无报警
  - `1`：常态系统提示 (仅日志或轻量介入)
  - `2`：黄色预警 (系统遇到轻微卡滞或非致命警示，拉响声光要求干预但不硬切电)
  - `3`：红色预警停机 (发生严重雷达撞击警告、禁区开门、心跳断联大面积掉线等，强制中断自动过程，要求硬切停机)
  

*(注：更为细致到引脚的 `Safety_Status_Word` 硬件安全干预及 `Device_Fault` 故障位域拆解，请前往详见第 7 章节。)*

## 6. 关键时序说明

基于设备工作说明与工艺流程图设计，封头检测工位在运行期间涉及多个环节的交互与闭环。系统所有基于 `Trig`_xxx 的业务均需遵循“触发 -> 执行 -> 结果 -> 应答 -> 清除”的标准流程，以确保交互同步一致性。

### 6.1 系统安全与设备在线监测时序
- **触发条件**：系统上电，循环执行（周期建议 50~100ms）。
- **前置条件**：PLC、工控机、网络连通正常。
- **动作顺序与交互**：
  1. PLC 定期刷新并读取基础硬件安全单元（如围栏门锁、光栅状态）。
  2. IPC（安全模块）保持 24h 不间断录像识别，循环判断是否有人进入区域；若有，通过 `Safety_Status_Word` 上报对应禁行位组合。
  3. IPC 端调用并回传吊装防碰和内壁防碰雷达的信号给 PLC。
  4. IPC 发送 `PLC_Heartbeat` 的周期心跳信号；若 PLC 检测到 3 秒无跳变则认定掉联。
- **异常逻辑**：无论处在何种模式，一旦出现人员侵入、设备掉线、碰撞前沿距离超门限，PLC 判定触发红色预警级别，锁停机械等主要动设备电源，发出声光报警并拦截自动状态机推进；软件持续弹窗直至排故复位。

### 6.2 测试初始化与多相机自检
- **触发条件**：人工上位机交互发下自检指令（且机械臂等位于原点安全位）。
- **流程与交互**：
  1. PLC 接到自检码 `Trig_SelfCheck`。
  2. IPC 收令后依序巡调 3D 相机外参、2D 解算以及跟踪标定校验项。
  3. IPC 将偏差量与合格判定通过对应的 `Ack_SelfCheck` 返回 PLC 设备故障字寄存器群 `Device_Fault` 某位域。
- **异常逻辑**：“自检”超阈值触发红色类故障，记录日志、禁止运行并提醒运维技师介入操作；合格后可开启允许自动运行位 `Safety_Status_Word.Bit7`。

### 6.3 上料工位判断与抓取点解算时序
- **触发条件**：已按下自动启动并具备放料准备，安全检测全部通过。
- **动作顺序与交互**：
  1. PLC 根据本地探测器判断上料口确实备好待传元件；同时扫描工位清空无占坑。
  2. PLC 写入必需的任务号、配方号、产品类型寄存器。
  3. PLC 发起 `Trig_LoadGrasp`=1 请求。
  4. IPC 捕捉触发侧沿，落盘 `Ack_LoadGrasp = 1`。
  5. IPC 指挥顶架 3D 相机执行拍照点云构建，解算提取 6 轴吸取基准法向，将结果送入 `Load_X~Load_Rz` 坐标缓冲组。
  6. 解算结果正常，IPC 设置 `Res_LoadGrasp=1`（成功），随即把握手推至 `Ack_LoadGrasp=2`。
  7. PLC 判断 Ack=2 后提取各轴位置，同时清零起始触发位 `Trig_LoadGrasp=0`。
  8. IPC 侦知 Trig=0 后回零 Ack，重置本状态机。
- **异常处理**：若 3D 审视结论为无料或超出抓取阈值，`Res_LoadGrasp` 即报相应错代码（如3:无料,4:位姿谬误），PLC 停止进给，拉响黄色告铃，请求人为接入复位判定。

### 6.4 上料搬运与放料扫描区协同
- **触发条件**：PLC 顺利拿到抓取解算坐标位，进入搬动周期。
- **前置条件**：路径上雷达未报侵入，转台清空且复零。
- **动作顺序**：
  1. PLC 根据目标做两步靠近抓取控制（机械臂抵近 -> 下向正方接触），充磁锁料。
  2. 机械臂起跳携物移向扫描工作转面区。
  3. 进行两步精准卸料投下（到达正上方 -> 贴贴退磁）。
  4. 放料完结，上下料机清让并回到待命泊车点。
- **断源及终止恢复机制**：若此阶段人为中断或停电掉链失磁，废除此件周期——由人工使用吊带运回上料点后再执行系统重置并清空数据复查。

### 6.5 扫描工位有无料与通孔位姿复校验
- **触发条件**：毛坯刚落实到小转台工位，进入正式检测前哨。
- **动作顺序与交互**：
  1. PLC 将带云台 2D 相机/扫描仪的“扫描机械臂”开入观测基点1号位。
  2. PLC 按序激活 `Trig_StationMaterialCheck`。
  3. IPC 拍照判断托盘是不是真的压了料；完成后向 PLC 交验 `Res=1(有)` `Ack=2`。
  4. 有料时 PLC 将其运往定点观测点2，紧接激活 `Trig_PoseCheck`。
  5. IPC 使用图像寻边算法分析中心通孔特征、确定夹持方势和位置约束，把偏差比回执填送 `Res` 答复 PLC。
  6. 数据接收及判定结束后，PLC 将双拉起触发复原。
- **异常处理**如果 `Res=3`（根本无料或未落正）或者位姿不可接纳，发声光报案并禁止推进核心扫描段；此时由人介入手工校核吊件或确认重新退磁卸走，不允许残件留阻。

### 6.6 分段扫描与多传感器联动执行时序
- **触发条件**：6.5 环节通过位姿考核放行开启多节点寻迹。
- **前置条件**：基于配方决定分几条路线与角度网格扫描该外体/内封缝隙组合。
- **动作顺序与交互**：
  1. PLC 编排本分段位置、通知转台驱动到相应法向、指示跟踪与 2D 到位就绪；同步下发值入当前段索引 `ScanSegmentIndex`。
  2. 到位后 PLC 致使 `Trig_ScanSegment`=1。
  3. IPC 回复 `Ack_ScanSegment=1`。
  4. IPC 驱动本段位姿下 2D 定向打光采集、稍微延迟后避光执行本段 3D 照面采集、再调用视觉跟踪坐标套系将其融合，封包暂存。
  5. 这一段保存即后，IPC 推向 `Ack_ScanSegment=2`。
  6. PLC 得回后置零触发；判断是否有下一段序号剩余。有则转向移动执行，无则退出执行扫描归零泊车。

### 6.7 综合 OCR 缺陷识别与 3D 解算时序
- **触发条件**：上阶段扫全轮廓返回空档准备阶段。
- **交互动作**：
  1. PLC 发送整体运算提报触发 `Trig_Inspection`。
  2. IPC 回答 `Ack_Inspection=1`，调集此前多套采集组，并发投给后发算法池：2D 进行 AI 表征编号 OCR 摘取、表面缺陷划识；3D 进行管口直径、高深厚度与坡角、接缝等计算汇演。
  3. IPC 根据多阈值筛算得总评价并附 NG 分项代码存入 `NG_Reason_Word0/1`，详细扩测数据（容积长度）填表，敲定判定结果 `Res_Inspection=1(正优)`/`其它=次废品`。
  4. 再发交 `Ack_Inspection=2` 放行终控端。
  5. PLC 提取结果判断入档，复 0 拉起源。

### 6.8 下料区满料判定与卸料点求解时序
- **触发条件**：整体检测已交卷，当前封头等待被遣离工作区域。
- **动作顺序与交互**：
  1. PLC 传出查询探信 `Trig_UnloadCalc` 以探明下游成品/暂留区空旷情况。
  2. IPC 推送 `Ack_UnloadCalc=1`。
  3. 利用大俯视场相机解定：判断这料垛区满没有；若无满找下妥安全的放堆点 6 轴定点位坐标。
  4. 传回判断参数（包括满还是空旷的状态词与对应坐标点），收尾为 `Ack=2`。
  5. PLC 辨识点位入驻清 0 通信信道进入倒序实操动作步骤。
- **满仓异常处理**：若 IPC 给的结果报出 `Res=3`（堆满无位），拉响故障音、黄色警备级卡着停工；通知场外拖车来搬空场区内堆件，重确认后才能起跑这最后落放行为。

### 6.9 下料执行落位复盘时序
- **最终执行段**：
  1. PLC 拿到确凿下放点。
  2. 使令下卸臂趋前，贴面转态充磁提拔走检完件，按安全迹划下落停置解磁（过程参照 6.4 之步法）。
  3. 把悬臂和所有动作盘复回其准生静默等待标点。
  4. 报告后级 MES 或自身系统本周期总循环结案，扫盲掉上下文内务标记量，敞开门禁准备下一组轮回件流程起效。

### 6.10 断线、急停中断与中途复始强制逻辑
- **突发停机场景**：例如任意时断电、人工大红拍下中断检测、防冲撞触达迫停机制触发等。
- **恢复与清理秩序交互**：
  1. **首要除杂**：复盘电力网络启动后，PLC 与 MES 控制台必须优先令 IPC **删清或废抛**该件暂缓存测算断包，不可继缝再用。
  2. **全面原点**：机构回返绝对零序。
  3. **空载探测防御**：为了免出碰撞失灾，恢复之前要在扫描端驱动下前先确认该小盘面上是否仍有遗留前件；PLC 必须触发一次判件操作。如真存在，拉下黄色警惕线和声光，禁止直接启动系统！——须由操作工人工摘挂、起转出界还原无残状态后，再撤消此戒备锁重起自动流序。

## 7. 设备状态与位域字典明细

为了提高系统对于复合状态和并发多因素的通信指标效率，部分报警、设备状态、以及复合原因类别采用 16位 (Word) 的按位 (Bit) 组合映射，其明细定义如下：

### 7.1 Device_Online_Word0 (IPC -> PLC)
用于展示 IPC 纳管之下相关重点硬件和服务模块的在线连通与存活状态：
- Bit0：上下料 3D 相机在线
- Bit1：扫描 3D 相机在线
- Bit2：2D 相机在线
- Bit3：AI 监控服务在线
- Bit4：视觉跟踪模块在线
- Bit5：读码/OCR 服务在线
- Bit6：基础算法服务在线
- Bit7：上层机器人通信在线
- Bit8：转台/滚轮架通信在线
- Bit9：声光报警器通信在线
- Bit10~15：预留

### 7.2 Device_Fault_Word0 (IPC -> PLC)
系统运行时 IPC 辖下核心传感器或算法模块层被动抛出的硬件、软件断言类错误：
- Bit0：上下料 3D 相机故障
- Bit1：扫描 3D 相机故障
- Bit2：2D 相机断流故障
- Bit3：AI 监控画面冻结或处理故障
- Bit4：视觉跟踪解算故障
- Bit5：OCR/读码引擎故障
- Bit6：通用算法模块崩溃或异常退出
- Bit7：数据库或文件结果写回层失败 (系统内部流转卡滞)
- Bit8：缓存数据存储耗尽异常 (落盘失败/数据库脱单)
- Bit9：任务时序缓存交叠/冲突异常 (指令队列错乱防呆)
- Bit10~15：预留扩展区

### 7.3 IPC_SafetyAction_Word (IPC -> PLC)
由 IPC 核心分析产生直接反馈给至 PLC 的软件级安全预警与防碰撞干预请求：
- Bit0：AI 监控报警 (发现人员越界)
- Bit1：吊装/运行防碰撞报警 (基于防碰雷达评估极距危险)
- Bit2：吊装辅助定位约束报警
- Bit3~15：预留（有待在电气图纸交付后确认：是否需要将 AI 报警以硬件接线形式直插下位机急停回路）

### 7.4 Safety_Status_Word (PLC -> IPC)
PLC 侧融合外围物理防线(光栅/锁)和上位机传达状态(`IPC_SafetyAction_Word`)汇聚后的**系统级全局安全禁行锁**，反传给 IPC 供全栈展示：
- Bit0：AI 监控告警锁定动作中 (承接自 IPC 预警判定)
- Bit1：安全光栅防线被触发阻断中
- Bit2：安全门锁正在物理打开
- Bit3：红色紧急按钮(E-Stop)硬触发中
- Bit4：吊装防撞极限预警强制锁定中
- Bit5：扫描段防碰撞强制锁定中
- Bit6：人员侵入干涉防线 (综合门/光/AI的终级禁止启动限制锁)
- Bit7：安全绿灯，全面允许自动运行 (当前没有任何安全警报方可置 1)
- Bit8~15：预留

### 7.5 SelfCheck_Fail_Word0 (IPC -> PLC)
配合时序 6.2，即相机与跟踪器等模组完成周期点准联动自检结果为 NG 时的附带病因定位：
- Bit0：3D 外参验证距离/偏差超限
- Bit1：视觉空间位姿跟踪拼接残差超限
- Bit2：2D 成像靶标比对不匹配异常
- Bit3：自检联络流程中受检设备意外掉线
- Bit4~15：预留

### 7.6 NG_Reason_Word0 (IPC -> PLC)
配合定性检测时序 6.7 回执，当产品被宣判为非良品(`Res=2/3`)时，提供并行的诊断根因多选项：
- Bit0：表面瑕疵/划痕/焊缝物理缺陷
- Bit1：几何尺寸参数/壁厚超差
- Bit2：无法捕捉验证 OCR 编号 (无码或严重覆污)
- Bit3：检测起始位置/孔形验证定准偏差离谱
- Bit4：高通扫描点云出现大面积空洞或反光散片补不齐
- Bit5~15：预留参数延展容差项

## 8. 扩展预留规划

### 8.1 控制命令区预留

- `40018~40019`：任务上下文与控制命令之间的缓冲预留，用于后续插入新的启动条件、配方联动或诊断控制量。
- `40031~40040`：控制命令、模式切换、联锁联动、诊断类扩展预留，优先满足后续新增命令位与过程控制字段的插入需求。

### 8.2 状态结果区预留

- `40111~40113`：设备在线/故障状态字之后的状态扩展区，适合后续增加设备细分在线状态、诊断摘要或状态确认字。
- `40175~40184`：在当前检测结果与安全干预区之后预留的连续结果扩展区，适合后续增加握手确认、诊断应答、测量补充值或工艺上下文回传字段。

### 8.3 结果扩展区预留

- `40173`：当前作为安全干预字使用，已保留较宽位域，便于未来扩展更多安全联锁来源。
- `40174`：安全干预字后的单点预留，用于后续接入补充安全状态、确认位或故障复位位。

## 11. 需 PLC 工程师重点核对项

1. Modbus 地址是否按 `40001` 方式展示，PLC 程序内部是否采用 `0` 基偏移
2. 32 位浮点字序是否采用 `CDAB`
3. 任务号是否需要拆为 2 个寄存器，或 PLC 仅保留 16 位流水号
4. `Ack=3` 是否保留为失败态，还是统一通过 `Res_xxx` 表示失败
5. 扫描是按“单段逐次触发”还是“整套路径一次触发”
6. OCR 编号结果是否需要直接回写字符，还是仅回写识别状态与编号索引
7. 下料区是否需要返回多个候选放置点
8. 红黄预警是否均由 PLC 驱动声光报警，还是黄色仅做软件提示
9. 自检是否允许“忽略并继续生产”，若允许需增加忽略确认位
