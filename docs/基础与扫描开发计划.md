# 封头检测工位：基础与扫描相机统合开发计划

## 1. 项目当前开发目标

当前阶段的核心业务目标是**优先攻退“扫描相机”模块**，以应对硬件条件限制及关键风险验证。然而，作为一个基于 Qt 开发的独立控制台应用（无 UI），若直接在此真空环境下强行开发相机模块，将导致后期代码耦合度过高、调试困难且无法与真正的 PLC 交互。

**因此，当前的综合目标为：**
建立一套**最小且必要**的底层基础支撑（通信、配置、日志、流程控制），随后在此地基上快速构建扫描相机业务体系。由于免除了本地数据库和图形化界面（UI）的沉重开发负担，基础模块的搭建将非常迅速，这为相机模块的稳定试运行提供了全方位的必要环境。

## 2. 开发策略结论

基于工程实施角度评估，**扫描相机模块不能（也不应）直接零基础开做**，必须优先拨出少量时间开发核心基础模块。具体策略如下：

- **程序架构定性**：基于 Qt Core（`QCoreApplication`）系统开发，依靠其强大的信号与槽机制调度任务，全盘舍弃纯 C++ STL 的杂糅写法。
- **必须先做的基础模块**：日志模块（无 UI 环境下的唯一调试窗口）、配置文件模块（提取硬编码）、Modbus TCP Client 模块（握手与触发的命门）。
- **同步/并行开发的基础模块**：业务状态机/流程中心控制模块模型（管理业务的流转）、内存数据缓存模块。
- **明确不开发的模块**：
  - **无数据库（No DB）**：数据持久化仅支持基础日志和关键分析记录文件输出，暂不引入 SQLite/MySQL 等。由于生命周期短，通过 Qt 内存容器（如 `QMap`, `QList`）暂存即可。
  - **无 UI 界面**：作为后台进程服务运行，剥离所有 `QWidget` 或 QML 渲染，节约系统开销。

## 3. 基础模块规划

为支撑扫描相机的四步握手（Trig->Ack=1->Res->Ack=2），以下基础模块划分为不同优先级。

### 3.1 配置文件管理模块（必须先做）
- **作用**：接管 IP 地址（PLC及相机）、端口、超时时长、重连次数等硬参数。
- **原由**：相机与 PLC 在现场调试中 IP 和端口修改频次极高，绝不能硬编码。
- **技术实现**：基于 Qt `QSettings` 封装，支持 INI 文件读取。
- **预期产出**：`ConfigManager` 单例模块。

### 3.2 诊断与日志模块（必须先做）
- **作用**：分级控制台输出及滚动文件写入（Info/Warn/Error/Debug）。
- **原由**：在无 UI 控制台程序中，日志是追踪设备触发、联调 Modbus 报文丢失、扫描超时的唯一“生命体征”。
- **技术实现**：通过重载 `qInstallMessageHandler` 接管 `qDebug/qWarning` 或引入 `QsLog` 级轻量框架。
- **预期产出**：全局 Logger。

### 3.3 Modbus 通信模块（必须先做）
- **作用**：维护与 PLC 的 TCP 长连接，暴露寄存器读写接口，封装底层报文。
- **原由**：扫描相机的启停依赖读取 `Trig_ScanStart` 信号，结果需要回写 `Res_ScanResult`。
- **技术实现**：基于 Qt 官方 `QModbusTcpClient` 进行封装，增加自动重连和读写队列，支持 `CDAB` 字节序转换逻辑。
- **预期产出**：`PlcCommunicator` 模块与寄存器值更新信号（Signals）。

### 3.4 状态机与流程控制模块（建议同步做）
- **作用**：调度“等待触发 -> 相机执行 -> 结果解算 -> 数据回填 -> 握手复位”的完整闭环。
- **原由**：相机的耗时计算必须采用异步（避免阻塞心跳），需要一套业务总控来编排这些异步节点。
- **技术实现**：利用 Qt 的 `QStateMachine` 或自定义的异步状态枚举配合信号槽。

### 3.5 结果缓存数据模块（并行/稍后做）
- **作用**：由于没有数据库，需实现带过期管理或批次管理的内存映射结构。
- **原由**：相机产生的点云、特征和缺陷详情需临时存放，直到装配完毕发给 PLC。
- **技术实现**：基于 Qt 隐式共享数据类（如 `QSharedDataPointer` 或常规自定义实体类）。

## 4. 扫描相机模块规划

在基础地基支撑上，扫描相机的规划如下：
- **功能目标**：实现相机驱动挂载、获取硬件触发/软触发后的图像采编、运算缺陷并生成规范结果码。
- **功能边界**：仅处理自身扫描任务，不越权涉足上下料和雷达安全区。
- **核心流程**：
  1. 监听 Modbus 收到的 `Trig_ScanStart` 上升沿（基于 Qt 信号槽机制）。
  2. 通知相机 SDK 抓图，工控机进入“处理中”状态。
  3. 获取数据流，调用具体算法核查缺陷。
  4. 将缺陷与类型包装并通知给 Modbus 模块回填。
- **输入输出**：入参为启动命令；出参为 `Res_ScanResult`、`NG_Reason_Word0` 报错字。
- **与基础模块依赖**：重度依赖 Logger 进行点云获取时长打点；重度依赖 Modbus 模块完成状态机流转。

## 5. 分阶段开发计划

### 第 1 阶段：基础能力最小集 
- **开发目标**：建立无界面的 Qt 后台服务框架并跑通通讯。
- **包含模块**：CoreApplication 启动框架、`QSettings` 封装、`Logger` 分级日志、`QModbusTcpClient` 通讯链路。
- **交付物**：能在控制台显示配置加载成功、并能与 PLC 模拟器（如 Modbus Slave）建立 TCP 连接和保持心跳（`IPC_Heartbeat`）的基础工程。
- **验证重点**：网络断开后能否自动重连；配置读写是否稳定；日志是否准确落盘。

### 第 2 阶段：扫描相机驱动 MVP 
- **开发目标**：打通与相机硬件的 SDK 握手。
- **包含模块**：扫描相机控制模块定义。
- **交付物**：能够脱离 PLC，仅通过程序内部测试指令呼叫相机采图，并将采图耗时或错误捕捉到日志中。
- **验证重点**：连续高频触发相机是否内存泄漏（无 DB 模式下内存尤为珍贵）；图像接收回调是否阻塞 Qt 主事件循环（Event Loop）。

### 第 3 阶段：业务流程联调与闭合 
- **开发目标**：将阶段 1 和阶段 2 拼装，执行真实协议流转。
- **包含模块**：流程控制中心模块、业务内存总线。
- **交付物**：能够通过外部 Modbus Poll 下发 1 触发，程序自动采图、分析、并严密按照 “回写结果 -> 回写状态 2”的顺序发还给外部工具。
- **验证重点**：四步握手的防抖与抗并发能力；各步骤的时序严谨度（遵守禁止在 `Trig!=0` 时清空结果区等原则）。

### 第 4 阶段：压力测试与内存调优阶段
- **开发目标**：应对无数据库架构带来的长期驻留风险。
- **交付物**：长时间运行可靠性检查补丁。
- **验证重点**：点云或大图片在处理完并响应给 PLC 后，其在内存中的主动销毁机制（如 `deleteLater` 等 Qt 智能内存管理机制）是否生效。

## 6. 开发顺序

由于采用全 Qt 开发：
1. **优先奠基**：日志模块与配置文件（最底层工具）。
2. **通讯握手跟进**：基于 Qt Network 机制完成 Modbus TCP（核心脉络）。
3. **相机单点攻关**：封装相机类并包装为其派发 Qt Signals（业务核心）。
4. **总装流程中心**：编写控制调度器，链接 Modbus 的信号与相机驱动的信号口（调度大脑）。

*理由：任何联调报错都需要日志支撑，因此日志必须第一；由于该工位完全依靠 Modbus 驱动运转，没通讯则业务无处下手，所以 Modbus 优先级高于相机业务封装。*

## 7. 风险与待确认项

- **无数据库的数据溯源风险**：如果发生现场纠纷，如何提供前 10 批次具体的点云与结果数据留存？是否需要基于直接存入磁盘 `PCD/JSON` 文件的降级替代方案？
- **硬件触发依赖问题**：扫描相机的飞拍触发信号是否由 PLC 以 Modbus 软命令下发？还是 PLC 以硬接线 DO 直接给相机？这极大地影响本程序的架构设计（若硬触发，本程序仅做“提取结果”角色；若软触发，本程序要做“下令发起”角色）。
- **阻塞与性能（单线程与多线程）**：由于采用 Qt 的事件驱动模型，如果扫描库/算法库运算时间过长且未部署在独立的 `QThread`，会导致 Modbus 响应超时断线。

## 8. 近期实施建议

- **第 1-2 天任务**：搭建基于 `QCoreApplication` 的 Qt 基础工程结构，配置好 Cmake/Qmake，编写独立的 `QsLog` 日志体系和 `INIManager` 配置文件系统。
- **第 3-5 天任务**：引入 Qt Modbus 库，搭建带断线重连的心跳系统，并能在本地与 Modbus 测试工具形成读写小循环。
- **暂时搁置**：复杂的算法业务检测逻辑与硬件设备的正式联调可以暂缓不写，先用 Mock（打桩）函数造假数据代替。
## 9. 封头检测工位：下一步核心业务开发与联调计划 (基于最新 6.11 时序)

### 阶段 1：对齐 6.11 时序机状态变迁更新
**目标：** 在现有 modules/flow_control/src/state_machine.cpp 的基础上，将流控动作切换到 6.11 所定义的最新精简时序，确保核心分支全覆盖。
- **任务 1.1：上料检定动作合并与参数整合**
  - 根据 6.11 时序，将“请求有无料”与“解算初始位姿/抓取点”结合。
  - 在状态机处理 Trig_StationMaterialCheck / Trig_PoseCheck 或 Trig_LoadGrasp 时，实现并行查询或依次调用模拟接口，确保能在一个应答周期内返回“有无料 (Res_StationMaterialCheck 或对应值)”以及“初始位姿 (Pose_X/Y/Z )”坐标值。
  - 测试 lt 分支：构造并回写代表“无料”或“位姿偏离”异常的 Res_xxx 及 Ack_xxx=3/2，并观测 PLC 端是否正确触发操作员报警。
- **任务 1.2：扫描过程中的 5~6 段 loop 循环接驳**
  - 使用 Qt 异步接口完善对 Trig_ScanSegment 的响应机制，维护基于 ScanSegmentIndex 的当前段落缓存。
  - 构造单段采集失败（如发生防碰撞干涉或硬件断流报警）并拉响 Res=2/5 等导致中止本件的操作。
- **任务 1.3：“满料判定”特化逻辑**
  - 在处理下料请求 Trig_UnloadCalc 时，除了返回 6 轴 Unload_X~Rz 坐标外，开发并调试对下料暂存区是否堆满的 Res=2(满料信号) 回写反馈。

### 阶段 2：真实相机与传感器控制层（API）接入
**目标：** 脱离 Modbus 固定占位符阶段，真实联动相机和读码器。
- **任务 2.1：初始化及相机服务封装 (modules/mech_eye)**
  - 实现真实 3D 相机的登录、参数设定和图像获取，提供统一或队列化的方法给到状态机调用。因为存在多台设备（上料、扫描区带云台相机），需要对每台相机对象独立管理。
- **任务 2.2：多设备硬件自检 (Trig_SelfCheck) 连通**
  - 新增所有挂载相机的状态查询指令，实现真实的心跳与自检校验（如果偏离阈值写回 SelfCheck_Fail_Word0 对位报错）。
- **任务 2.3：后台耗时任务剥离 (至关重要)**
  - “本段执行扫描”及相机 SDK 传输可能耗时数秒，必须通过 QThread、QtConcurrent 或信号槽机制投入后台工作者线程（Worker），防止阻塞 IPC 的 Master Modbus Polling (50~100ms)，确保期间 IPC_Heartbeat 正常更迭，防止网络掉线。

### 阶段 3：真实视觉算法/跟踪模块（CV / 3D 处理）整合
**目标：** 实现 modules/tracking 下的核心定位算法与拼接逻辑接入，用真实的数据结构驱动总解算。
- **任务 3.1：扫描各段点云/位姿的高效缓冲管理**
  - 解决无数据库下（No DB）点云计算资源的生命周期管理。维护在 5~6 段 loop 循环中采集的点云/图源临时映射（使用如 QSharedDataPointer 或标准容器的智能指针封装），确保随时能被合并调用。
- **任务 3.2：执行拼接与综合缺陷判定 (Trig_Inspection)**
  - 把 6.11 图中“拼接并整体解算”环节与最终算法处理引擎连接（OCR字符获取、尺寸比对、深度判定等）。
  - 根据返回的判定参数，填入 NG_Reason_Word0 的特征位及返回整体检测结果 (Res_Inspection=1(OK)/2(NG)) 并执行缓存（点云、内存）的智能销毁，防止内存泄漏。
- **任务 3.3：断线、急停清除残存 (Trig_ResultReset 与 异常处理机制)**
  - 遇到异常报警 (Device_Fault_Word0 或 触发 Safety_Status_Word) 的打断后，必须强力复原状态机并从缓存区清除未执行完的上件残存数据，等待 PLC 发送的 Trig_ResultReset 或再次触发判件操作重新步入正常工序。

### 阶段 4：联机真实 PLC 与下厂测试
**目标：** 基于前期模拟验证和算法替换完成度，进行真实设备综合压测。
- **任务 4.1：系统联调与配置校准**
  - 修改 pp/config.ini 至现场 PLC 真实 IP 和参数（包含重连间隔超时常数，确保符合 6.11 内的超时设限）。
  - 启动工控机并在现场完整跑通以下单件周期：取料 -> 放料安全监测 -> 扫描位段触发 1-6 -> 计算综合评价 -> 送至下料栈堆 -> 解除。
- **任务 4.2：边界与压力验证**
  - **超频率打断**：在执行相机采样或算法合并计算中期，手动触发安全红门/雷达侵入预警，检验系统能否被稳态中断和清理环境。
  - **连续运行压测**：循环工作 4-8 小时，观察日志（logs/ 下）有无超时记录及任务堆叠报错，确认内存分配正常无遗漏销毁。
